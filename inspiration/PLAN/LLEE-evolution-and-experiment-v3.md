## 一、进化路径的价值判断

### 核心洞察

Evolutionary-thinking.txt 揭示了一个关键事实：**当前实验计划的评估指标（CLIP 分数、填充率、证据校准度）本身就是进化算法的适应度函数。** 实验计划和自进化计划不是两件事——前者是后者的手动版本。

```
当前计划（人类驱动）：
  人类设计 Schema → 跑实验 → 看指标 → 人类修改 Schema → 再跑

自进化计划（算法驱动）：
  初始 Schema → 跑实验 → 计算适应度 → 遗传算法变异 Schema → 再跑

唯一的区别：谁在做"选择"这一步。
```

### 这意味着什么？

1. **不需要额外的"自进化实验"**——只需要在 Phase 2 之后，把人类的"看指标→改 Schema"替换为脚本的"读指标→变异 Schema"
2. **进化不是远期梦想，是当前实验的自然延伸**
3. **但 Demo 阶段不做进化**——先证明"人类设计的 Schema 有效"，再证明"Schema 可以自动优化"

### 联想到的其它内容

进化路径的思考还可以延伸到：

**A. Schema 的"物种分化"**

不同类型的叙事（动作戏 vs 对话戏 vs 环境描写）可能需要不同的 Schema 粒度。进化算法可以自然产生"物种分化"——一套 Schema 分裂为多个特化变体，每个变体适应不同的叙事生态位。

**B. 证据分级规则的进化**

当前的证据分级规则（TEXT 置信度上限 1.0，PREDICTED 上限 0.4）是人工设定的。这些阈值本身也可以被进化搜索优化——什么样的阈值组合能最大化人类评估者的"忠实度"评分？

**C. 与 JEPA 的真正交汇点**

JEPA 学习连续表征，LLEE 进化离散符号。两者的交汇点是：**用 JEPA 学到的表征空间作为 LLEE 进化的"地形图"**——符号在表征空间中的位置决定了它们的语义距离，进化算法在这个地形上搜索最优的符号分布。这是"逆 LLEE"方向的一个具体化。

## 二、回应五个疑问——修正实验设计

### 疑问 1 回应：语料差异是否足够大？

**这是最关键的前置验证。必须在正式实验前确认。**

```
解决方案：预实验（Pre-study）

Step 1：从两个语料中各取 5 个段落，确认它们描述的是同一叙事事件
  - 如果不是 → 需要人工创建对齐语料（同一事件的精简/丰富两种写法）
  - 如果是 → 继续

Step 2：量化信息密度差异
  - 词数比：丰富版 / 精简版 的平均词数比（目标 > 2.0）
  - 感官词密度：每 100 词中涉及视觉/听觉/触觉/嗅觉的词数
  - 实体提及密度：每 100 词中提及的可渲染实体数量
  
  如果这些指标差异不显著 → 语料不合格，需要重新准备

Step 3：用 3 个段落试跑 Parser，检查填充率差异
  - 如果丰富版填充率 < 精简版的 1.5 倍 → 差异不够，需要加大语料差距
```

**对计划的修改**：在 Phase 0 中增加"语料验证"步骤，作为 Phase 1 的门控条件。

### 疑问 2 回应：CLIP 分数对精简语料是否公平？

**确实不公平。CLIP 天然偏好信息密集的文本。**

```
解决方案：双度量体系

度量 A — CLIP 对齐分数（自动化，有偏但可复现）
  用途：大规模自动评估，作为筛选工具
  局限：对精简语料不公平，H2 可能是平凡的

度量 B — 归一化忠实度分数（半自动，公平但昂贵）
  定义：渲染结果中"正确呈现的信息量" / "源文本中可渲染的信息总量"
  
  例：
    精简文本："阿拉丁走进洞穴"
    可渲染信息：{人物=阿拉丁, 动作=走, 目标=洞穴} = 3 个信息单元
    渲染结果正确呈现了 3 个 → 归一化忠实度 = 3/3 = 1.0
    
    丰富文本："阿拉丁弯腰穿过低矮石缝，潮湿空气扑面，远处铜灯发蓝光"
    可渲染信息：{人物, 姿态=弯腰, 空间=低矮石缝, 触觉=潮湿, 光源=铜灯, 光色=蓝} = 6 个
    渲染结果正确呈现了 4 个 → 归一化忠实度 = 4/6 = 0.67

  这个度量对两种语料都公平：精简语料可以得满分（因为分母小），
  丰富语料更难得满分（因为分母大）。

度量 C — 人类直接评分（Phase 3，最终裁判）
  作为 CLIP 和归一化忠实度的校准基准
```

**对计划的修改**：
- Phase 1 增加"信息单元标注"——为每个段落标注可渲染信息单元数量
- Phase 2 同时报告 CLIP 分数和归一化忠实度
- 论文中讨论 CLIP 的偏差，用归一化忠实度作为补充

### 疑问 3 回应：WSM 状态漂移率的 ground truth

**接受建议：降级为定性观察。**

```
修改后的方案：

定量指标（保留但简化）：
  - 每 10 个段落设置一个"检查点"
  - 只在检查点处标注完整的累积状态（而非每段都标注）
  - 50 段序列 = 5 个检查点 = 5 次人工标注
  - 漂移率 = 检查点处 WSM 状态与人工标注的字段级不一致比例

定性观察（新增）：
  - 在 UI 中实时显示 WSM 当前状态
  - 人工抽查 3 个"叙事转折点"（场景切换、时间跳跃、新角色出场）
  - 记录观察笔记：WSM 是否正确处理了状态重置？
```

### 疑问 4 回应：Three.js 渲染质量是否足够？

**接受建议：聚焦环境而非角色。**

```
修改后的渲染策略：

Phase 2 的 Three.js 渲染聚焦于：
  ✓ 光照（方向、色温、强度、阴影）— 差异最明显
  ✓ 色调/氛围（后处理：色彩分级、雾效、对比度）— 情绪传达的主要载体
  ✓ 空间（场景几何：洞穴/沙漠/室内）— 环境类型的直接体现
  ✓ 粒子效果（灰尘、水汽、火星）— 增加沉浸感
  
  ✗ 角色面部表情 — Three.js 做不好，且不是核心验证点
  ✗ 角色动画 — 同上
  ✗ 精细材质 — PBR 参数差异在低多边形模型上不明显

角色处理方案：
  - 用剪影/轮廓表示角色位置和姿态
  - 或用 2D 立绘叠加在 3D 环境上（视觉小说的经典做法）
  - 情绪通过环境光照和音频传达，而非面部表情

增加 Phase 2.5 — 渲染质量感知测试（n=5）：
  "你觉得这个画面的质量是否足以让你判断它是否符合文本描述？"
  如果 < 4/5 人说"是" → 调整渲染策略后再进入 Phase 3
```

### 疑问 5（隐性假设）回应：LLM 结构化输出的稳定性

**这是最大的风险。必须在一切之前验证。**

```
新增 Phase 0.5 — Prompt 稳定性验证

目标：确认 Claude API + JSON Schema 约束能稳定输出合规的 LLEE Delta

方法：
  1. 选 5 个代表性段落（覆盖所有语言类型）
  2. 每个段落跑 20 次（共 100 次调用）
  3. 记录：
     - Schema 合规率（目标：100%）
     - 证据分级一致性（同一段落 20 次输出的证据等级是否稳定）
     - 情绪标签一致性（同一段落 20 次输出的情绪标签是否稳定）
  
  通过标准：
    - Schema 合规率 = 100%（结构化输出约束应保证这一点）
    - 证据分级一致性 > 80%（允许 CONTEXTUAL/UNDEFINED 边界有波动）
    - 情绪标签一致性 > 90%（核心判断应稳定）
  
  如果不通过：
    - 迭代 prompt 直到通过
    - 如果 10 轮迭代仍不通过 → 考虑换 LLM 或降低 Schema 复杂度
    - 记录所有迭代过程 → 论文素材

  时间预算：3-5 天（在 Phase 0 和 Phase 1 之间）
```

## 五、关键决策点（Gate）

```
Gate 0 → Phase 1：语料验证通过？信息密度差异显著？
Gate 0.5 → Phase 1：Prompt 稳定性通过？合规率 100%？
Gate 1 → Phase 2：Parser 保真度数据是否支持核心假设？
  - 如果 IEI 泄漏率 > 20% → 回退修改 prompt
  - 如果填充率差异不显著 → 回退检查语料
Gate 2 → Phase 3：渲染质量感知测试通过？
  - 如果不通过 → 调整渲染策略或改为定性研究
Gate 3 → Phase 4：论文核心数据是否足够？
  - 如果 Phase 1-3 数据已足够发表 → Phase 4 作为后续工作
  - 如果需要更强的贡献 → Phase 4 的进化实验作为论文亮点
```

## 六、时间线总览

```
Week 1-2    Phase 0     语料验证 + Schema 定义 + 技术选型
Week 2-3    Phase 0.5   Prompt 稳定性验证（3-5 天）
Week 3-5    Phase 1     Parser 管道 + 保真度实验 ← 第一个可发布结果
Week 5-8    Phase 2     渲染 + 自动化评估 + 演示视频
Week 8-11   Phase 3     人类评估 + 论文撰写
Week 12+    Phase 4     进化实验 + 扩展（可选）
```

## 七、与进化路径的关系

```
Phase 0-3 证明：人类设计的 LLEE Schema 有效
Phase 4 证明：Schema 可以通过进化算法自动优化
两者合在一起证明：LLEE 不仅是一个静态的工程框架，它是一个可进化的符号系统

这正是 LLEE 区别于 FactTrack / CRANE / ChatUSD 的独特贡献：
  - FactTrack 追踪事实，不进化
  - CRANE 约束解码，不进化
  - ChatUSD 生成 USD，不进化
  - LLEE 的 Schema 本身是进化的对象——这是一个活的语言，不是一个死的格式
```

---

## 八、一句话总结

**LLEE 的实验计划不是在验证一个静态系统，而是在为一个可进化的符号语言建立第一组适应度基准。** Phase 1-3 是基准测量，Phase 4 是进化启动。两者用同一套指标，同一套管道，同一套代码——区别只在于谁在做"选择"：人还是算法。
LLEE不是在“优化一个prompt”，而是在**建立AI生成内容与确定性渲染之间的信任边界**。IEI抑制、证据分级、WSM差分编码——这些具体技术都是为了实现这个边界而做的工程决策。评估数据不是目的，而是证明“这个边界确实被建立起来了”的证据。
