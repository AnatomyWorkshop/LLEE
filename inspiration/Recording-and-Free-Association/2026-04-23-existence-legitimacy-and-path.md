# LLEE 开发笔记 — 存在合法性、静态与动态、以及我们到底在做什么

> 2026-04-23

---

## 一、LLEE 的存在合法性

### 最初的目的是什么？

最初的目的很简单：**让 AI 从文本生成画面时，不要编造文本里没有的东西。**

一个人读到"阿拉丁走进洞穴"，脑海中浮现的画面因人而异——有人想象的洞穴是潮湿的，有人想象的是干燥的。这些差异是合理的，因为文本没有说。但如果 AI 渲染器给阿拉丁加上了"恐惧的表情"，而文本里阿拉丁没有表现出恐惧——这就是幻觉，是注入，是 IEI。

LLEE 的存在理由就是这一条：**在 LLM 和渲染器之间建立一个可审计的事实锚点，确保渲染不违背文本。**

### 这个理由今天还成立吗？

成立。而且比最初更成立。

Phase 1 Round 1（GLM）和 Round 2（DeepSeek Chat）的数据一致证明：

| 指标 | GLM (Round 1) | DeepSeek Chat (Round 2) |
|------|---------------|------------------------|
| LLEE Full 填充率 | 52% | 68% |
| LLEE Full 环境填充率 | 36% | 60% |
| IEI 泄漏率（所有结构化条件） | 0 | 0 |
| Condition C（二元证据）Schema 合规率 | 0/12 | 0/12 |

两个不同的 LLM，同一套 Schema 约束，IEI 都是零。这不是巧合，是约束层在起作用。

DeepSeek Chat 的填充率显著高于 GLM（68% vs 52%），说明 LLEE 的约束层不会压制更强模型的能力——更强的模型在同样的约束下提取更多信息，但仍然不注入幻觉。**约束层是天花板，不是地板。**

### 但 Constantly-review 的质疑呢？

质疑的核心是：**LLEE 只是一个"索引"，不是一个"描述"。它不生成画面，只约束画面。最终渲染结果的大部分是由渲染器的默认值决定的，而非由 LLEE 的符号决定的。**

这个质疑完全正确。但它不否定 LLEE 的存在合法性——它精确地定义了 LLEE 的边界。

类比：交通信号灯不生成任何交通流量。它只约束车辆在什么时候可以通过路口。没有人会说"交通信号灯没有存在合法性，因为它不生成交通"。信号灯的价值恰恰在于它不生成任何东西——它只确保通过路口的车辆不会撞在一起。

LLEE 就是叙事渲染管线中的交通信号灯。它的价值不在于它生成了什么，而在于它阻止了什么。

## 二、静态到动态：我们是否要完全改变路径？

### 当前的静态架构

LLEE v0.6 是纯静态的：
- WSM 接收 Delta，应用 Delta，输出新状态
- 它不预测下一个状态
- 它不学习任何转移函数
- 它的"动态"只有衰减——而衰减是预设的指数衰减，不是学习出来的

### Constantly-review 指出的问题

> "渲染器从静态符号中无法推断出动态过程。光线的传播需要光的路径，声音的回响需要声波的反射，情绪的表现需要面部肌肉的连续运动。"

这是对的。但这个问题的答案不是"LLEE 应该变成动态系统"，而是"LLEE 应该承认自己是静态的，并设计好与动态渲染器的接口"。

### 不应该完全改变路径

理由：

1. **静态是 LLEE 的核心优势，不是缺陷。** 静态意味着可审计、可复现、可解释。如果 LLEE 变成动态预测系统，它就失去了"事实锚点"的身份——因为预测本身就是一种幻觉。

2. **动态是渲染器的责任。** Three.js 知道如何做体积光、粒子系统、布料模拟。LLEE 不需要重新发明这些。LLEE 只需要告诉 Three.js："这里有一束光，从裂缝到铜灯，色温 3200K，强度 0.7。" Three.js 自己会计算光的路径、散射、阴影。

3. **Phase 4 的预测器是一个可选的扩展，不是路径改变。** 预测器的输出是 PREDICTED 级别的证据（置信度上限 0.4），它不会覆盖文本事实，只是在文本沉默时提供低置信度的填充。这是在静态骨架上加一层薄薄的动态皮肤，不是把骨架换成动态的。

### 但 v0.7 需要承认的事情

v0.7 论文必须在 Introduction 中明确声明：

> LLEE 是一个约束系统，不是生成系统。它不生成画面或声音；它约束下游渲染器在生成视听内容时的自由度，以确保生成结果不违背文本中可验证的事实。LLEE 输出的是渲染的"锚点"（anchors），而非渲染的"内容"（content）。锚点之间的空间由渲染器的默认资产和物理模型填充。

这不是自贬。这是划定 LLEE 真正不可替代的价值。

## 三、丰富的语言渲染流动世界究竟要如何实现？

这是最根本的问题。让我们从头想。

### "丰富的语言渲染流动世界"意味着什么？

它意味着：一段文字——比如"阿拉丁弯腰穿过低矮石缝，潮湿空气扑面，远处铜灯发蓝光"——被转化为一个你可以走进去的、有光影流动、有声音回响、有空气湿度的三维世界。

这需要三层能力：

```
第一层：理解（文本 → 结构化语义）
  "潮湿空气" → humidity=0.8, particle_type=water_vapor
  "铜灯发蓝光" → light_source=lamp, material=copper_oxide, color_temp=8000K

第二层：约束（结构化语义 → 渲染参数边界）
  humidity=0.8 → fog_density ∈ [0.3, 0.5], particle_count ∈ [200, 500]
  color_temp=8000K → light_color ∈ [#a0c0ff, #b0d0ff]

第三层：生成（渲染参数 → 画面/声音/触觉）
  fog_density=0.4 → 体积雾着色器
  light_color=#a8c8ff → 点光源 + 体积光散射
  humidity=0.8 → 环境音：滴水声，混响 wet=0.6
```

**LLEE 当前只做第一层的一部分（理解），和第二层的一小部分（约束）。第三层完全不做。**

### 缺失的关键环节

1. **第一层的缺失：环境信息提取不足。** Phase 1 数据显示环境填充率只有 36%-60%。文本中 40%-64% 的环境信息被 Parser 遗漏了。这是 LLM attention 偏向角色/情绪的结果。

2. **第二层的缺失：从语义到渲染参数的映射是空的。** `render_intensity()` 是一个分段线性函数，输出一个无单位标量。它不知道 humidity=0.8 应该映射到什么雾效密度，也不知道 tension=0.7 应该映射到什么色温偏移。

3. **第一层和第二层之间的缺失：没有"语义锚点"的概念。** LLEE 输出的 `scene_type="cave"` 对渲染器来说只是一个字符串标签。渲染器需要的是：洞穴的几何参数（高度、宽度、弯曲度）、材质参数（岩石类型、湿度、苔藓覆盖率）、光学参数（环境光遮蔽、反射率）。这些参数不在 LLEE 的 Schema 中。

### v0.7 必须补充的地方

**补充 1：render_intensity 的 sigmoid 升级 + 语义锚点**

当前：
```python
def render_intensity(self) -> float:
    if self.confidence > 0.8: return 1.0
    elif self.confidence > 0.5: return 0.3 + (self.confidence - 0.5) * 1.0
    ...
```

v0.7 目标：
```python
def render_intensity(self, target_param: str = "generic") -> float:
    """Sigmoid mapping with per-parameter semantic anchors."""
    alpha, beta = RENDER_ANCHORS.get(target_param, (10.0, 0.5))
    return 1.0 / (1.0 + math.exp(-alpha * (self.confidence - beta)))

RENDER_ANCHORS = {
    "fog_density":    (8.0, 0.4),   # 低阈值，快速响应
    "color_temp_shift": (12.0, 0.6), # 高阈值，需要强证据
    "reverb_wet":     (6.0, 0.3),   # 低阈值，环境音敏感
    "particle_count": (10.0, 0.5),  # 中等阈值
}
```

这不是 PBR 级别的物理模型。但它为每个渲染参数提供了一个可解释的、可调的、可微的映射函数。Phase 4 可以用人类评分数据梯度优化这些 (α, β) 参数。

**补充 2：UNDEFINED 的子类型拆分**

```
UNDEFINED_ALEATORIC:  文本本身没有信息 → 渲染器用中性默认值，不可覆盖
UNDEFINED_EPISTEMIC:  文本有信息但 Parser 漏了 → 渲染器可以做低强度上下文插值
```

这将 IEI 的根源精确锁定为"混淆两种不确定性"。

**补充 3：衰减率的敏感性数据**

今天的理论分析结果：

```
衰减率变体        ATMOSPHERIC 持续回合    CONTEXTUAL 持续回合
baseline (0.7/0.9)    5.0 turns              21.9 turns
+20% (0.84/1.0)      10.3 turns             ∞ (永不衰减!)
-20% (0.56/0.72)     3.1 turns              7.0 turns
```

关键发现：
- ATMOSPHERIC 对 ±20% 扰动的响应是 2-3x 变化 → 中等敏感
- CONTEXTUAL 在 +20% 时达到 1.0（永不衰减）→ 高度敏感，存在临界点
- 这意味着 CONTEXTUAL 的衰减率 0.9 处于一个危险的区域——稍微高一点就失去衰减功能

v0.7 论文应该报告这个结果，并在 Discussion 中讨论：是否应该将 CONTEXTUAL 衰减率降低到 0.85 以远离临界点？

**补充 4：跨模型对比数据**

今天的 DeepSeek Chat 数据 vs 之前的 GLM 数据：

```
指标                    GLM-4.5-air    DeepSeek Chat    差异
LLEE Full 填充率        52%            68%              +16pp
LLEE Full 环境填充率    36%            60%              +24pp
Condition D 填充率      57%            68%              +11pp
Condition D 环境填充率  ~36%           55%              +19pp
IEI 泄漏（所有条件）    0              0                =
Condition C Schema 合规  0/12           0/12             =
```

关键发现：
- 更强的模型在同样的约束下提取更多信息，但 IEI 仍然为零
- 环境填充率的提升最显著（+24pp），说明 DeepSeek Chat 的 attention 分布比 GLM 更均匀
- Condition C（二元证据）在两个模型上都完全不合规，说明六类型证据框架对 Schema 合规性是必要的

## 四、LLEE 做的工作有必要吗？

### 如果没有 LLEE，会怎样？

没有 LLEE，叙事渲染管线是这样的：

```
文本 → LLM → "阿拉丁看起来很害怕，洞穴阴暗潮湿" → 渲染器 → 画面
```

问题：
- LLM 说"阿拉丁看起来很害怕"——但文本里阿拉丁没有表现出恐惧。这是 IEI。
- LLM 说"洞穴阴暗潮湿"——但文本只说了"洞穴"，没说阴暗也没说潮湿。这是环境幻觉。
- 渲染器无法区分"文本说的"和"LLM 编的"。

### 有了 LLEE，管线变成：

```
文本 → Parser(+LLEE Schema) → Delta(带证据分级) → WSM → 渲染器
```

- Parser 输出 emotion=UNDEFINED（因为文本没说阿拉丁害怕）→ 渲染器不加恐惧表情
- Parser 输出 scene_type="cave", evidence=EXPLICIT → 渲染器知道"洞穴"是文本说的
- Parser 输出 humidity=UNDEFINED → 渲染器用默认值，不编造"潮湿"

**LLEE 的工作是必要的，因为没有它，渲染器无法区分事实和幻觉。**

### 但这个工作是否"足够"？

不够。LLEE 目前只解决了"不要编造"的问题，没有解决"如何丰富"的问题。

一个只有事实锚点的渲染，看起来会很"空"——因为文本通常只提供 30%-60% 的环境信息，剩下的 40%-70% 是 UNDEFINED。渲染器在 UNDEFINED 区域只能用默认值，结果就是"正确但贫乏"的画面。

这就是 Constantly-review 说的"LLEE 描述不了任何画面"的真正含义——不是说 LLEE 的输出是错的，而是说 LLEE 的输出太少了，不足以驱动一个丰富的渲染。

### 解决路径

```
短期（v0.7）：
  - 提高环境填充率（更好的 prompt + 更强的模型）
  - render_intensity sigmoid + 语义锚点 → 让已有的信息驱动更丰富的渲染
  - UNDEFINED 子类型 → 让渲染器在 EPISTEMIC 区域做可控的插值

中期（Phase 2-3）：
  - 渲染器的"默认资产库"本身就是丰富的（Three.js 的洞穴场景预设）
  - LLEE 的锚点约束渲染器"不要偏离文本"，但不阻止渲染器"在文本允许的范围内丰富"
  - 人类评估验证：用户是否觉得"约束下的丰富"比"无约束的幻觉"更好？

长期（Phase 4）：
  - 上下文预测器：在 UNDEFINED 区域提供低置信度的填充
  - 进化算法：优化 Schema 结构，让更多信息被提取
  - JEPA 融合：用连续向量补充离散符号的信息缺失
```

## 五、一句话总结

LLEE 的存在合法性不在于它能描述多丰富的世界，而在于它能阻止多少幻觉进入渲染管线。它是叙事渲染的交通信号灯——不生成交通，但确保交通不撞车。v0.7 的任务是让这个信号灯有数学规格书（M1-M4），让它与渲染器的接口有物理语义（sigmoid + 语义锚点），并用跨模型数据证明它在不同 LLM 上都有效。路径不需要从静态改为动态——静态是 LLEE 的核心优势。动态是渲染器的事。
