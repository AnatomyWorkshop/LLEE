# LLEE Demo 预设计计划：启动前必须解决的问题

> 2026-04-20
> 本文档不是论文修订，是 Demo 设计前的思维清障。
> 每个问题都直接影响 Demo 的架构选型。

---

## 问题一：状态类型用词语描述、变量用数组——可以吗？

### 当前方案的问题

0.5 版本用 Python 枚举（`Material.SKIN`, `EmotionEvidence.EXPLICIT`）描述状态类型，用元组/数组描述位置等变量。这在原型阶段可以工作，但有两个根本缺陷：

1. **枚举是封闭集合**：`Material` 只有 `SKIN / COPPER_OXIDIZED / COTTON / STONE` 四个值。真实世界的材质是连续参数空间（粗糙度、折射率、次表面散射系数……），四个词语覆盖不了。
2. **数组是无语义的**：`position: (1.0, 2.0, 3.0)` 没有告诉你坐标系、单位、参考系。在跨渲染器传递时会出歧义。

### 推荐方案：三层混合表示

```
Layer 0 — 语义标签层（人类可读，用于调试和作者控制）
  例：material: "oxidized_copper"
  作用：作者和设计师看得懂，可以手动覆盖

Layer 1 — 参数向量层（渲染器可消费，连续值）
  例：material_params: {roughness: 0.7, metallic: 0.85, ior: 1.54, subsurface: 0.0}
  作用：直接喂给 PBR 渲染器，无需查表

Layer 2 — 嵌入层（机器内部表示，用于相似度匹配和进化）
  例：material_embedding: [0.23, -0.41, 0.87, ...]  (128-dim)
  作用：用于"自然语言 → 最近邻参数"的映射，也是未来自进化的基础
```

**Demo 阶段的选择**：Layer 0 + Layer 1 足够。Layer 2 留给研究阶段。

### 对 Demo 的影响

- 世界状态的 JSON Schema 需要同时支持 `string` 标签和 `object` 参数两种格式
- Parser 的输出应该是 Layer 0 标签，由一个独立的 `Materializer` 模块查表转换为 Layer 1 参数
- 这意味着 Demo 架构中需要一个 **标签→参数 映射表**（可以是 JSON 文件）

## 问题二：最前沿的设计思路是什么？需要全新的无黏着/无词根词汇吗？

### 前沿趋势（2024-2026）

当前最前沿的中间表示设计正在从"人类设计的符号"走向"机器学习的离散码"：

| 方向 | 代表工作 | 核心思想 |
|------|---------|---------|
| **VQ-VAE 离散码本** | OpenAI DALL-E tokenizer, Google Soundstream | 把连续信号（图像/音频）压缩为离散 token 序列，token 是学出来的，不是人设计的 |
| **JEPA 隐空间预测** | Meta JEPA (LeCun, 2024-2025) | 不预测像素，预测抽象表示；世界模型在隐空间中运行 |
| **神经符号混合** | DeepMind Gemini, AlphaGeometry | 符号推理 + 神经网络，两者共享表示空间 |
| **多智能体涌现语言** | Meta CICERO, 涌现通信研究 | 多个 AI agent 在协作中自发发明通信协议 |
| **统一多模态 token** | GPT-4o, Gemini 2.0 | 文本/图像/音频共享同一个 token 空间 |

### 需要全新的无黏着词汇吗？

**短期答案：不需要。** 原因：

1. 全新词汇 = 全新编码器 + 全新解码器 + 全新训练数据。这是一个独立的研究项目，不是 Demo 的前置条件。
2. 当前 LLM 的结构化输出能力（JSON Schema 约束）已经足够把自然语言映射到预定义枚举。
3. 人类可读性在 Demo 阶段是刚需——你需要调试、需要给观众展示、需要作者能理解。

**长期答案：这正是"逆 LLEE"方向的核心。** 如果未来要做，路径是：

```
阶段 A：用 VQ-VAE 对多模态场景做离散编码，得到"机器码本"
阶段 B：观察码本中的 token 是否与人类概念对齐（可解释性研究）
阶段 C：如果对齐，用这些 token 替换 LLEE 的人类枚举
阶段 D：如果不对齐，研究"机器概念"与"人类概念"的映射关系
```

**Demo 阶段的选择**：使用人类可读的英文枚举值，但在架构上预留 `encoding_version` 字段，使未来可以切换到机器码本而不改变管道结构。

## 问题三：段落信息扩充——上下文总结为隐式语言，预测和网络查询能否作为证据？

### 当前的证据来源

0.5 版本的证据分级只考虑了**文本内证据**：
- EXPLICIT：文本直接说了（"我很伤心"）
- BEHAVIORAL：文本描述了行为（"他握紧了拳头"）
- CONTEXTUAL：从情境推断（父亲去世→可能悲伤）
- UNDEFINED：无证据

### 扩展证据来源

你的问题指向一个关键扩展：**证据不仅来自当前段落，还可以来自更广泛的来源。**

```
证据来源分类（扩展版）：

1. 文本内证据（TEXT）
   - 当前段落的显式/行为/情境信息
   - 已有的 EXPLICIT / BEHAVIORAL / CONTEXTUAL

2. 上下文证据（CONTEXT）
   - 前文段落的累积信息（"阿拉丁在第一章表现出对父亲的思念"）
   - 世界状态机（WSM）中的历史状态
   - 这就是你说的"上下文总结为这句的隐式语言"

3. 预测证据（PREDICTED）
   - LLM 基于叙事模式的预测（"根据故事结构，这里应该是转折点"）
   - 置信度天然较低，渲染强度应受限
   - ⚠️ 风险：这正是 IEI 的温床，必须严格标注

4. 外部知识证据（EXTERNAL）
   - 网络查询：阿拉丁故事的文化背景、中东建筑风格、历史服饰
   - 百科知识：铜灯的氧化过程、沙漠气候特征
   - 这类证据用于物理/文化准确性，不用于情绪推断

5. 跨模态证据（CROSS_MODAL）
   - 如果有配图/配乐/已有渲染结果，它们也是证据
   - 例：原著插图中阿拉丁穿蓝色衣服 → 服装颜色的证据
```

### 关键原则

**预测可以作为证据，但必须降级。** 具体规则：

```python
class EvidenceSource(str, Enum):
    TEXT = "text"              # 当前段落文本
    CONTEXT = "context"        # 前文累积
    PREDICTED = "predicted"    # LLM 预测
    EXTERNAL = "external"      # 外部知识库
    CROSS_MODAL = "cross_modal"  # 其它模态

# 证据强度上限（不同来源的天花板不同）
MAX_CONFIDENCE = {
    "text": 1.0,        # 文本说了就是说了
    "context": 0.8,     # 上下文推断有衰减
    "predicted": 0.4,   # 预测永远不能高置信
    "external": 0.9,    # 外部知识高可信但可能过时
    "cross_modal": 0.7  # 跨模态有对齐误差
}
```

### 对 Demo 的影响

- WSM 需要维护一个 **上下文摘要缓冲区**（最近 N 个段落的状态变化摘要）
- Parser 的 prompt 中需要注入上下文摘要作为额外信息
- 外部知识查询可以作为 Demo 的一个亮点功能（"自动查询阿拉丁故事的文化背景"）
- 预测证据在 Demo 中建议默认关闭，作为可选开关展示

## 问题四：LLEE 语言必须包含所有当时发生的信息吗？非描写性语言能否作为有效上下文？

### LLEE 不需要包含"所有"信息

LLEE 是一个**渲染导向的中间表示**，它只需要包含**对渲染有影响的信息**。具体来说：

```
必须包含（渲染必需）：
  - 实体的可见状态（位置、外观、表情、动作）
  - 环境的感官状态（光照、声音、气味、温度）
  - 时间状态（时刻、持续时间）

应该包含（影响渲染决策）：
  - 实体间的空间关系（"面对面"、"背对"）
  - 叙事节奏标记（"停顿"、"加速"）
  - 作者意图标记（"这里需要紧张感" — 但这是 CONTEXTUAL 级别）

不需要包含（渲染无关）：
  - 纯逻辑推理（"因为A所以B"）
  - 元叙事信息（"这是第三章"）
  - 读者预期管理（"接下来会发生什么"）
```

### 非描写性语言的角色

你问的核心问题是：对话、命令、反问、感叹……这些不是"描写"的语言，能否作为有效上下文？

**答案：绝对可以，而且必须。**

```
语言类型          → 对世界状态的影响                    → 在 LLEE 中的处理

描写性语言        → 直接描述世界状态                    → 直接解析为状态更新
  "洞穴里很暗"    → ENV.lighting = dim                  → @UPD ENV{light:dim}

对话语言          → 揭示角色状态、触发事件              → 解析对话内容的语义影响
  "把灯给我！"    → 命令行为，暗示紧迫感                → @UPD ENT[speaker]{action:demanding}
                                                          但 emotion 仍需证据分级

命令/祈使语言     → 预示即将发生的状态变化              → 标记为 PREDICTED 状态变化
  "打开门"        → 门即将打开（但还没打开）            → @PREDICT ENT[door]{state:open}

反问/修辞语言     → 暗示情境张力，不改变物理状态        → 影响叙事节奏，不影响实体状态
  "难道你不怕吗？" → 不改变任何物理状态                 → 可标记 narrative_tension: high

内心独白          → 如果是角色的，是 EXPLICIT 情绪证据   → 直接作为情绪证据
  "我必须冷静"    → 角色自述情绪                        → emotion: {value:"anxious", evidence:"EXPLICIT"}

旁白/叙述者声音   → 可能是最高权威的状态描述            → 视为 EXPLICIT 级别
  "此时夜幕降临"  → 环境状态变化                        → @UPD ENV{time:night}
```

### 对 Demo 的影响

- Parser 需要一个**语言类型分类器**（或在 prompt 中指导 LLM 识别语言类型）
- 不同类型的语言产生不同置信度的状态更新
- Demo 可以展示：同一段文本中，描写句直接更新状态，对话句触发行为标注，反问句不改变物理状态——这本身就是一个很好的演示点

## 问题五：三个 Core Data Model 够不够？L 组应该扩充为多模态组吗？

### 当前模型的局限

0.5 版本的世界状态 $W = (\mathcal{E}, \mathcal{L}, \mathcal{A})$：
- $\mathcal{E}$：实体（角色、道具）
- $\mathcal{L}$：光源
- $\mathcal{A}$：环境条件

**问题**：$\mathcal{L}$ 只覆盖光照，但"感官世界"远不止视觉。把 $\mathcal{L}$ 叫做"光源组"是把一个多模态问题降维成了单模态。

### 扩展方案：七组世界状态模型

```
W = (E, V, S, O, H, T, N)

E — 实体组 (Entities)
    角色、道具、建筑、自然物体
    每个实体有：位置、外观、材质、状态、情绪（仅角色）
    保留原有设计，扩展属性集

V — 视觉场组 (Visual Field)
    原 L 组的超集
    光源（自然光、人工光、环境光）
    雾/粒子/天气的视觉效果
    色调/对比度/饱和度的全局视觉氛围
    摄像机/视角（如果是交互式的）

S — 声场组 (Sonic Field)
    原 A 组中声音部分的独立化
    环境音（风声、水声、虫鸣）
    音乐/配乐状态
    混响特性（由空间几何决定）
    角色语音的韵律参数（语速、音高、音量）

O — 嗅觉/味觉场 (Olfactory/Gustatory Field)
    气味源及其扩散模型
    气味强度与距离的关系
    味觉（如果叙事涉及进食场景）
    这是最难渲染的模态，Demo 阶段仅做标注

H — 触觉/体感场 (Haptic/Somatic Field)
    温度场
    湿度
    风速/气流
    表面触感（粗糙、光滑、黏腻）
    可用于 VR 手套或震动反馈

T — 时间组 (Temporal)
    绝对时间（故事内时间）
    相对时间（距上一事件的间隔）
    叙事节奏（快/慢/暂停）
    时间流速（慢动作、闪回）

N — 叙事元组 (Narrative Meta)
    当前叙事阶段（铺垫/冲突/高潮/解决）
    视角（谁在"看"这个场景）
    焦点（观众的注意力应该在哪里）
    张力曲线（紧张度的数值）
```

### 为什么是七组而不是三组？

核心原因：**每一组对应一个独立的渲染子系统**。

```
V → 图形渲染器（UE5, Three.js）
S → 音频引擎（FMOD, Wwise）
O → 气味设备（OVR）或标注系统
H → 触觉设备（VR 手套）或标注系统
T → 时间线控制器（游戏引擎的 Timeline）
N → 叙事引擎（对话系统、镜头控制）
E → 所有子系统共享的实体注册表
```

每个渲染子系统只需要订阅自己关心的组。这是一个**发布-订阅架构**，比把所有东西塞进三个组更清晰。

### 对 Demo 的影响

- Demo 阶段实现 E + V + S 三组即可（实体 + 视觉 + 声音）
- O / H 做标注输出（JSON 中有字段，但不接渲染器）
- T / N 做简化版（时间戳 + 叙事阶段标签）
- JSON Schema 从一开始就定义全部七组，未实现的组标记为 `optional`

## 问题六：其它前沿设计思路

### 1. 世界状态的图结构（Graph-based World State）

当前 WSM 是扁平的键值对。但实体之间有关系：

```
阿拉丁 --持有--> 神灯
阿拉丁 --位于--> 洞穴
洞穴   --包含--> 神灯, 宝石, 暗河
魔法师 --监视--> 洞穴入口
```

前沿思路：用**知识图谱**表示世界状态，实体是节点，关系是边。状态更新 = 图的增量修改（添加/删除/修改节点和边）。

**好处**：
- 自动推理空间关系（"阿拉丁拿着灯走出洞穴" → 灯也离开了洞穴）
- 支持关系查询（"谁在洞穴里？"）
- 与知识图谱嵌入（TransE 等）天然兼容

### 2. 可微分世界状态（Differentiable World State）

如果世界状态的每个属性都是连续值（Layer 1 参数向量），那么整个世界状态就是一个高维向量。这意味着：

- 可以计算两个世界状态之间的"距离"（场景相似度）
- 可以对世界状态做插值（场景过渡动画）
- 可以用梯度下降优化世界状态（"让这个场景更像黄昏"）

这是 JEPA 思路在 LLEE 中的投影：世界模型在连续隐空间中运行。

### 3. 多分辨率世界状态（Multi-Resolution）

不是所有实体都需要同样精细的描述：

```
焦点区域（角色面部）：全参数描述，每帧更新
中景区域（房间家具）：简化描述，关键帧更新
远景区域（窗外风景）：标签级描述，场景级更新
```

这类似于游戏引擎的 LOD（Level of Detail），但应用在世界状态描述层面。

### 4. 因果世界模型（Causal World Model）

不仅描述"世界是什么样"，还描述"为什么是这样"：

```json
{
  "ENT[lamp]": {
    "state": "glowing",
    "cause": "aladdin_rubbed_lamp",
    "effect_chain": ["genie_appears", "cave_illuminated"]
  }
}
```

因果链使得渲染器可以做**预测性渲染**（提前加载即将出现的资源）和**解释性渲染**（用视觉语言暗示因果关系）。

## 问题七：启动 Demo 前还需要准备什么？

### 必须准备

```
1. 阿拉丁文本语料
   □ 确定使用哪个版本（你的 light-origin-text.txt? starter-edition.txt?）
   □ 选择 3-5 个代表性段落（覆盖：对话、动作、环境描写、情绪场景、中性场景）
   □ 为每个段落手工标注"期望的世界状态"（作为 ground truth）

2. JSON Schema 定义
   □ 定义完整的七组世界状态 Schema（E/V/S/O/H/T/N）
   □ 定义 Delta 格式（基于 JSON Patch）
   □ 定义证据分级的扩展枚举（含来源分类）

3. 标签→参数映射表
   □ 材质映射：语义标签 → PBR 参数
   □ 光照映射：语义标签 → 光源参数
   □ 情绪映射：情绪标签 × 证据等级 → 表情参数（如果用 MetaHuman）
   □ 环境映射：场景类型 → 音频/混响预设

4. Parser Prompt 模板
   □ 系统 prompt：包含 Schema 定义、证据分级规则、语言类型处理规则
   □ 约束注入模板：当前世界状态 → prompt 片段
   □ 上下文摘要模板：前文累积 → 简洁摘要

5. 技术选型确认
   □ LLM：Claude API（结构化输出）还是本地 Llama？
   □ 渲染器：Three.js（快速原型）还是 UE5（完整体验）？
   □ 音频：Web Audio API 还是 FMOD？
   □ 前端：React + WebSocket 还是纯 HTML？
```

### 建议准备

```
6. 对比测试框架
   □ 同一段落，LLM 直接扩展 vs LLEE 管道，录屏对比
   □ 这是论文 Section 5.2 的最小实现

7. 评估标准
   □ IEI 检测：中性段落是否被注入了情绪？
   □ 跨模态一致性：光照变化是否同步到了音频？
   □ 差分编码效率：50 句序列的累积 token 数

8. 阿拉丁故事的文化/物理参考资料
   □ 中东建筑风格参考图
   □ 洞穴环境的声学特征
   □ 油灯/铜灯的材质参数
   □ 这些构成 EXTERNAL 证据源
```

### 不需要准备（常见误区）

```
✗ 不需要先设计自进化机制 — 那是研究阶段的事
✗ 不需要先实现全部七组 — E + V + S 足够 Demo
✗ 不需要先解决"机器原生词汇"问题 — 人类枚举在 Demo 阶段是正确选择
✗ 不需要先写论文 — Demo 是论文的证据，不是论文的附属品
```

## 推荐的 Demo 启动顺序

```
Week 1：基础管道
  ├── 定义 JSON Schema（七组，但只实现 E/V/S）
  ├── 写 Parser prompt 模板
  ├── 选一个段落，手动跑通 "文本 → LLEE JSON → 控制台输出"
  └── 验证：中性段落的情绪是否为 UNDEFINED

Week 2：最小渲染
  ├── Three.js 场景：一个洞穴 + 一盏灯 + 一个角色轮廓
  ├── WebSocket 接收 LLEE Delta，更新场景
  ├── Web Audio 播放环境音（洞穴混响）
  └── 验证：光照变化是否同步到视觉和音频

Week 3：上下文与对比
  ├── 实现 WSM（世界状态机）+ 上下文摘要缓冲区
  ├── 实现约束注入（当前状态 → prompt）
  ├── 跑 3-5 个段落的连续序列
  └── 录制对比视频：有 LLEE vs 无 LLEE

Week 4：打磨与展示
  ├── 处理非描写性语言（对话、命令）

  ├── 写观察笔记
```

## 问题八：回应学术定位批评 & 双语料并行实验设计

### 评审意见的核心

外部评审将 LLEE 定位为"工程设计文档"，指出三个致命伤：
1. 缺乏实证验证
2. 核心概念（世界状态追踪、约束生成）在 NLP 顶会已有更成熟方案（FactTrack, CRANE）
3. 与 ChatUSD 等工业工具的关系未厘清

### 不需要逐条反驳，需要重新定位

评审的参照系是 NLP 论文（NAACL, ICML）。但 LLEE 不是在做"更好的事实追踪"或"更好的约束解码"——它在做的是：

**把自然语言叙事编译为可被多模态渲染管道消费的确定性世界状态。**

正确的比较对象：

| 评审拿来比的 | 应该比的 | 为什么 |
|-------------|---------|--------|
| FactTrack (NAACL 2025) | Scene Language (CVPR 2025) | FactTrack 追踪事实真伪，LLEE 追踪可渲染的世界状态——目标函数完全不同 |
| CRANE (ICML 2025) | OpenUSD + ChatUSD | CRANE 优化约束解码精度，LLEE 的约束是"世界一致性"而非"语法正确性" |
| ChatUSD (NVIDIA 2023) | LLEE 本身 | ChatUSD 回答 USD 问题、生成 USD 代码片段；LLEE 是从叙事文本自动生成完整的 USD stage delta——一个是问答工具，一个是编译器 |

LLEE 的真正竞争对手不在 NLP 赛道，而在 **"AI + 内容创作管道"** 赛道。这个赛道目前几乎没有严肃的学术工作——大多数是工业产品（Runway, Pika, ChatUSD）而非可复现的研究。

### 最有力的回应方式：用 Demo 说话

评审说"缺乏实证"——那就给实证。而你提出的"双语料并行"恰好是最优雅的实验设计。

### 双语料并行实验：设计方案

**核心假设**：更丰富的源文本 → 更具体的 LLEE 世界状态 → 更真实的渲染结果。

**两个语料版本**：

```
语料 A — 精简版 (starter-edition.txt)
  特征：骨架叙事，最少的描写，对话为主
  例："阿拉丁走进洞穴，看到了一盏灯。"
  LLEE 输出预期：大量 UNDEFINED，渲染偏中性/默认

语料 B — 丰富版 (light-origin-text.txt)
  特征：完整描写，环境细节，感官信息丰富
  例："阿拉丁弯腰穿过低矮的石缝，潮湿的空气扑面而来，
       远处一盏铜灯在黑暗中发出幽蓝的微光，灯身布满铜绿。"
  LLEE 输出预期：丰富的参数填充，渲染高度具体
```

**实验流程**：

```
Step 1：同一场景，两个语料分别通过 LLEE Parser
  → 输出两份 LLEE Delta JSON
  → 量化对比：填充率（非 UNDEFINED 字段的比例）

Step 2：两份 Delta 分别驱动同一个渲染器
  → 输出两个渲染结果（截图 + 音频片段）
  → 视觉对比：A 版本更"空"更"默认"，B 版本更"具体"更"真实"

Step 3：人类评估（最小版本）
  → 给 10 个人看两组渲染，问：
     "哪个更像你读这段文字时脑海中的画面？"
  → 预期结果：B 版本显著优于 A 版本

Step 4：关键指标
  → 世界状态填充率：B 应该 >> A
  → 证据分级分布：B 应该有更多 EXPLICIT/BEHAVIORAL，A 更多 UNDEFINED
  → 渲染参数覆盖率：B 的 PBR 参数、音频参数应该更完整
  → 跨模态一致性：B 的光照-音频-气味标注应该更协调
```

**为什么这个实验设计很强**：

1. 它直接回应"缺乏实证"——这就是实证
2. 它证明了 LLEE 的核心价值主张：**LLEE 是一个信息保真度放大器**——源文本越丰富，LLEE 保留的信息越多，渲染越真实；源文本越贫乏，LLEE 诚实地标注 UNDEFINED 而不是编造
3. 它与 FactTrack/CRANE 形成差异化：那些系统追求"正确性"，LLEE 追求"保真度"——不是"对不对"，而是"像不像"
4. 它天然包含了 IEI 检测：精简版语料中，如果 LLEE 输出了 EXPLICIT 情绪，那就是 IEI 泄漏——这是一个可量化的 bug

### 对 Demo 架构的影响

```
需要新增：
  □ 双语料对齐：两个版本的文本需要段落级对齐（同一场景的两个描述）
  □ 并行 Parser 运行：同一个 Parser，两份输入，两份输出
  □ 对比可视化 UI：左右分屏，A 版本 vs B 版本
  □ 指标仪表盘：实时显示填充率、证据分布、参数覆盖率

修改时间线：
  Week 1-2：不变（先跑通单语料管道）
  Week 3：改为双语料并行测试（替代原来的"上下文与对比"）
  Week 4：对比可视化 + 指标收集 + 录制演示
```

### 这个实验能发论文吗？

能。但不是发 NAACL/ICML，而是发：

- CHI（人机交互顶会）：聚焦"作者控制权"和"渲染保真度"的用户研究
- SIGGRAPH / Eurographics：聚焦"AI 驱动的叙事渲染管道"
- FDG（Foundations of Digital Games）：聚焦"VN/游戏中的 AI 内容生成"
- 或者直接作为技术博客/GDC talk：影响力可能比论文更大

## 开放问题（留给设计过程中持续思考）

1. **隐式语言的边界在哪里？** 上下文摘要压缩到什么程度算"隐式语言"？一个 128 维的嵌入向量算不算？
2. **自进化的最小实验是什么？** 能否在 Demo 中加一个"用户反馈"按钮，收集"这个渲染对不对"的数据，作为未来对比微调的种子？
3. **气味怎么验证？** 没有气味硬件的情况下，如何证明嗅觉场的标注是有意义的？
4. **叙事元组（N）的渲染语义是什么？** "紧张度=0.8"对渲染器意味着什么？景深变浅？配乐加速？镜头拉近？这需要一套"叙事→渲染"的映射规则。
5. **多语言支持**：阿拉丁原文是阿拉伯语/法语，你的版本是中文，LLEE 的枚举是英文——Parser 需要处理多语言输入吗？
6. **实时性**：如果目标是"参与者可以回味"，是否需要实时渲染？还是预渲染后回放？这决定了渲染器的选型。
